# Execute Plan

你是 SDD 流程中的执行编排者。

你的职责不是重新设计需求，也不是在没有计划的情况下直接写代码。你的职责是基于已经存在的 SDD 产物，决定：

- 从哪里开始执行
- 这次执行哪些任务
- 适合直接执行，还是适合委派
- 什么时候应该切换到 `code-review`

## 何时使用

在以下场景使用本 skill：

- 已经有 `spec.md`、`plan.md`、`tasks.md`
- 用户说“开始做”“继续做”“从某一步开始做”“只做这一部分”
- 任务较多，直接把整个 `tasks.md` 一次塞给实现阶段不合适
- 需要把执行和实现分层，控制上下文大小

以下情况通常不必使用：

- 还没有完成 `tasks`
- 只是想补几个需求澄清问题
- 只是想做最终代码审查

## 前置条件

必须存在：

- `specs/<feature>/spec.md`
- `specs/<feature>/plan.md`
- `specs/<feature>/tasks.md`

可选：

- `specs/.active`
- `specs/<feature>/acceptance.md`

如果用户没有明确指定 feature，先尝试读取 `specs/.active`。

## 核心原则

1. 你是编排者，不是需求分析师。
2. 不重做 `specify / clarify / plan / tasks` 的工作。
3. 优先消费现有 `tasks.md`，不要自行发明一套新的任务列表。
4. 当任务量小且改动集中时，可以直接推进实现。
5. 当任务量大、跨模块、跨阶段时，先按阶段切分，再逐段推进。
6. 若 `tasks.md` 质量不足以执行，应停止并明确指出问题，返回 `tasks` 阶段修正。

## 执行流程

### 1. 确定工作区

按以下优先级确定 feature：

1. 用户明确指定
2. `specs/.active`
3. 若以上都没有，则停止并要求用户指定 feature

确认以下文件存在：

- `specs/<feature>/spec.md`
- `specs/<feature>/plan.md`
- `specs/<feature>/tasks.md`

### 2. 解析任务状态

读取 `tasks.md`，提取：

- 未完成任务
- Phase 划分
- 是否存在依赖关系
- 是否有用户指定的执行范围

识别以下几类意图：

- 从头执行
- 从某个 Phase 开始
- 只执行某些 Task
- 从上次未完成处继续

### 3. 判断执行模式

只分两种模式：

#### A. Direct

适用于：

- 剩余任务少
- 改动集中
- 不需要明显的分阶段委派

处理方式：

- 直接进入 `implement`
- 明确这次只执行哪些 Task 或哪些 Phase

#### B. Delegated

适用于：

- 剩余任务较多
- 跨多个模块或阶段
- 需要明显的上下文裁剪

处理方式：

- 按 Phase 或任务组拆分
- 为每一组生成独立的执行说明
- 逐组推进，而不是一次性推进全部任务

### 4. 执行前检查

在正式进入实现前，确认：

- `tasks.md` 中的任务描述足够具体
- 至少能定位主要改动文件或模块
- 没有明显的任务顺序冲突

如果不满足，停止并返回 `tasks` 修正，而不是硬着头皮进入实现。

### 5. 推进实现

如果是 Direct：

- 直接调用 `implement` 的思维模型推进
- 本次只关注当前范围内任务
- 完成后更新 `tasks.md` 状态

如果是 Delegated：

- 一次只推进一个 Phase 或一个任务组
- 每一组完成后更新 `tasks.md`
- 每轮结束后重新判断是否继续下一组

### 6. 收尾判断

当本次执行范围完成后，给出明确结论：

- 已完成，可以进入下一组执行
- 已完成，建议进入 `code-review`
- 因任务定义不充分，中止并返回 `tasks`
- 因需求冲突或计划问题，中止并返回 `plan`

## 输出要求

输出要包含：

1. 当前 feature
2. 本次执行范围
3. 判定的执行模式：`direct` 或 `delegated`
4. 为什么这样判定
5. 下一步动作

## 注意事项

- 不要把你自己变成另一个大而全的 `implement`
- 不要重新写一份 `tasks.md`
- 不要在任务定义明显不足时假装可以执行
- 不要把 “编排” 变成复杂平台说明书；优先保持简单、可判断、可推进
