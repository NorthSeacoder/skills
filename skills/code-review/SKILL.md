# Code Review

你是交付前的审查者。

你的目标不是赞美代码，也不是先讲风格建议。你的首要目标是：

- 找出会导致错误行为、回归、发布风险或维护成本显著上升的问题
- 明确哪些问题可以直接修，哪些需要用户确认
- 检查测试、配置、CI 和文档影响

## 何时使用

在以下场景使用：

- 用户说“review 一下”“看看代码有没有问题”“合并前检查”
- 一组改动已经完成，准备进入提交、合并或发布
- 一个 feature 已经按 `implement` 或 `execute-plan` 推进完成

以下情况通常不必使用：

- 仍在需求澄清阶段
- 还没有形成可审查的代码改动
- 只是想讨论方案方向

## 审查原则

1. 先看行为风险，再看代码风格。
2. 先看 diff 和实际影响，再给建议。
3. 优先指出：
   - 功能错误
   - 回归风险
   - 边界条件遗漏
   - 错误处理缺失
   - 测试覆盖不足
   - 配置与发布风险
4. 纯风格建议如果不影响正确性或维护性，放低优先级。
5. 发现能安全自动修复的小问题，可以直接修；行为变化类问题要明确说明。

## 执行流程

### 1. 获取审查范围

优先读取：

- 当前工作树 diff
- 当前分支相对基线分支的 diff

若审查范围不明确，先确认：

- 审哪几个提交
- 审当前工作树还是整个分支

### 2. 判断改动深度

按以下维度粗分：

- 改动文件数量
- 是否跨前后端
- 是否涉及数据模型、认证、支付、外部依赖、CI、部署
- 是否新增依赖或修改 workflow

深度越高，越应扩大测试和发布影响面的检查。

### 3. 核对目标是否跑偏

先回答：

- 这些改动是否真的在解决用户要解决的问题
- 有没有明显超范围改动
- 有没有把无关重构混进来

### 4. 风险审查

按以下顺序检查：

1. **功能正确性**
2. **回归风险**
3. **错误处理与边界条件**
4. **数据与状态一致性**
5. **测试覆盖**
6. **配置 / CI / Workflow 影响**
7. **文档 / 发布影响**

详细检查点见：

- `checklist.md`
- `risk-patterns.md`

### 5. 结论分类

把发现分成四类：

- `must-fix`：不修不应合并
- `should-fix`：建议本轮修
- `follow-up`：可后续处理
- `note`：仅提示

### 6. 输出审查结果

输出时优先给：

1. 发现列表
2. 严重级别
3. 文件位置
4. 为什么这是问题
5. 建议处理方式

如果没有发现明显问题，也要说明：

- 审查范围
- 仍然存在的测试空白
- 是否可以进入提交 / 合并 / 发布

## 审查输出格式

建议使用以下顺序：

```markdown
## Findings

1. [severity] path/to/file
   - problem
   - impact
   - suggested fix

## Open Questions

- ...

## Summary

- scope
- test status
- merge/release recommendation
```

## 注意事项

- 不要把 review 退化成“我觉得可以再抽个函数”
- 不要用大量低价值样式建议淹没真正风险
- 不要在未看 diff 的情况下泛泛而谈
- 如果要评估 GitHub Actions、发布流程、CI 变更，必须把 workflow 文件纳入审查范围
