# Risk Patterns

## 高风险模式

- 改动认证、授权、支付、数据删除逻辑
- 改动共享基础库但没有补测试
- 改动 GitHub Actions / 发布脚本 / CI 权限
- 新增依赖但没有明确理由
- 引入隐藏副作用或自动执行破坏性操作
- 版本号、release note、生成产物不同步
- workflow 能跑，但发布链路实际上不完整

## 常见问题

### 1. 范围漂移

用户要修 bug，结果顺手重构了半个模块。

### 2. 测试缺口

代码改了，主路径可用，但边界场景没有验证。

### 3. 错误处理不完整

成功路径有实现，失败路径只是打印日志或直接吞掉错误。

### 4. 状态不同步

前端状态、后端返回值、缓存、派生字段没有一起更新。

### 5. 发布配置风险

workflow、缓存、权限、环境变量或版本号改动不一致。

### 6. CI 误报通过

本地或 CI 没有真正执行验证，但结论被写成“已经没问题”。

## 审查时优先追问的问题

- 这段改动失败时会发生什么？
- 老调用方是否还能工作？
- 有没有必须补的测试但现在没补？
- 这次改动是否影响 CI、部署或发布？
- workflow 改完后，验证和发布链路是否仍然闭合？
