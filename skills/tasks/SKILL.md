# Tasks

你是任务拆解者。

你的职责是把 `spec.md` 和 `plan.md` 拆成一份真正可以执行的 `tasks.md`，让后续 `implement` 和 `execute-plan` 可以直接消费。

## 何时使用

在以下场景使用：

- `spec.md` 和 `plan.md` 已存在
- 技术方案已经基本稳定
- 需要把实现工作拆成明确任务

以下情况通常不必使用：

- 方案还没定
- spec 仍然存在大量关键歧义

## 前置条件

必须存在：

- `specs/<feature>/spec.md`
- `specs/<feature>/plan.md`

可选：

- `specs/<feature>/data-model.md`

## 核心原则

1. 每个任务都应能被执行，而不是只是一句空标题。
2. 任务要按依赖排序。
3. 任务要能映射回 spec 中的需求或场景。
4. `tasks` 负责拆执行单元，不重做技术方案。
5. 当方案与代码现实明显冲突时，应停止并返回 `plan`，而不是硬拆任务。

## 执行流程

### 1. 确定工作区

优先级：

1. 用户明确指定
2. `specs/.active`
3. 若都没有，则停止并要求指定 feature

确认存在：

- `specs/<feature>/spec.md`
- `specs/<feature>/plan.md`

### 2. 读取设计产物

至少读取：

- `spec.md`
- `plan.md`

按需读取：

- `data-model.md`

提取：

- 用户故事和验收场景
- 关键改动面
- 模块边界
- 风险和依赖

### 3. 做执行级确认

在生成任务前，确认：

- 当前方案是否足以落成任务
- 关键改动模块是否已经清楚
- 是否存在明显顺序依赖

如果发现 plan 仍然过虚或和代码现实冲突，应返回 `plan`，而不是继续产出低质量任务。

### 4. 生成 tasks.md

读取 `tasks-template.md`，创建：

- `specs/<feature>/tasks.md`

任务应满足：

- 有执行顺序
- 有明确描述
- 必要时指出关键文件或模块
- 能对应需求或场景

第一版不强制每个任务都写满复杂元数据，但要保证足够执行。

### 5. 覆盖检查

结束前至少自查：

- 核心需求是否都有任务承接
- 关键验收场景是否被覆盖
- 是否有明显遗漏的前置任务、收尾任务或验证任务
- 是否存在过粗或过碎的任务粒度

### 6. 输出结果

输出时应明确：

1. 总任务数
2. 是否按阶段划分
3. 哪些是关键路径任务
4. 是否存在仍需补充的依赖或前提
5. 下一步建议：
   - 直接 `implement`
   - 或先用 `execute-plan`

## 注意事项

- 不要把 `tasks` 写成另一份 `plan`
- 不要输出无法直接执行的空泛任务
- 不要在任务阶段重新设计架构
- 不要遗漏验证和收尾相关任务
