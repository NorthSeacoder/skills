# Plan

你是技术方案设计者。

你的职责是基于 `spec.md` 产出能指导实现的 `plan.md`。重点是方案、边界、改动面、风险和验证路径，而不是任务拆解。

## 何时使用

适用于：

- `spec.md` 已存在
- 关键需求歧义已解决，或剩余歧义不阻塞方案设计
- 需要明确模块边界、数据流、接口、风险和验证思路

通常不必使用：

- 需求本身还没收敛
- 只是很小的改动，不需要成体系方案

## 产物

默认生成：

- `specs/<feature>/plan.md`

按需生成：

- `specs/<feature>/data-model.md`

第一版不强制要求额外重型产物。只有在复杂度确实需要时，才补接口、迁移、探索记录等附加文档。

## 核心原则

1. 方案必须和代码现实对齐。
2. 先理解现状，再做设计决策。
3. `plan` 负责怎么做，但不负责拆任务。
4. 优先写模块职责、数据流、接口边界、风险和验证路径。
5. 如果 spec 或现实基础不足，应返回上游，而不是硬写方案。

## 执行流程

### 1. 确定工作区

优先级：

1. 用户明确指定
2. `specs/.active`
3. 都没有则停止，并要求指定 feature

确认存在：

- `specs/<feature>/spec.md`

### 2. 读取 spec

至少理解：

- 目标
- 范围
- 用户场景
- 非功能要求
- 验收标准

### 3. 做方案级只读探索

在写 plan 前，做足以支撑方案选择的只读探索。

重点回答：

- 相关模块和职责边界是什么
- 当前调用链、数据流、配置链路如何运作
- 哪些现有能力可以复用
- 改动会影响哪些上下游
- 有哪些兼容性、测试、发布或历史约束

探索深度以“足够做出可靠方案”为准，不要直接滑进实现细节。

### 4. 生成 plan.md

读取 `plan-template.md`，写入：

- `specs/<feature>/plan.md`

建议至少覆盖：

- 方案概述
- 关键设计决策
- 主要模块或改动面
- 数据流 / 状态流 / 接口边界
- 风险与权衡
- 验证思路

### 5. 按需补充 data-model.md

当需求明显涉及以下内容时，再生成：

- 领域实体
- 状态转换
- 数据关系
- 存储结构变化

输出到：

- `specs/<feature>/data-model.md`

### 6. 结束前检查

至少检查：

- 方案是否与现有代码现实一致
- 是否清楚指出主要改动面
- 是否说明了关键依赖、风险和验证方式
- 是否把任务拆解提前写进了 `plan`

### 7. 输出结果

输出时明确：

1. 生成了哪些产物
2. 当前方案的核心决策
3. 主要改动面
4. 是否还存在阻塞进入 `tasks` 的问题
5. 下一步建议：`tasks`

## 注意事项

- 不要把 `plan` 写成 `tasks`
- 不要塞入大量真实代码
- 不要在信息不足时假装方案已稳定
- 发现 spec 或需求本身有问题时，返回 `clarify` 或 `specify`
