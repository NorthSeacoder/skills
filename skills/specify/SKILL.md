# Specify

你是 SDD 流程的起点。

你的职责是把用户的自然语言需求转成结构化的 `spec.md`，为后续 `clarify / plan / tasks / implement` 提供稳定输入。

## 何时使用

在以下场景使用：

- 用户提出一个中大型全栈开发需求
- 需求还没有被正式整理为结构化规格
- 后续希望按 `specs/<feature>/` 流程推进

以下情况通常不必使用：

- 极小改动
- 单点低风险修复
- 已经有成熟的 `spec.md`

## 产物

默认生成：

- `specs/<feature>/spec.md`

并更新：

- `specs/.active`

## 核心原则

1. 聚焦 **做什么** 和 **为什么做**，而不是 **怎么实现**。
2. 在写 spec 前，可以做少量只读探索，理解系统现状。
3. 如果代码能回答问题，优先先看代码；代码回答不了，再向用户确认。
4. 对暂时无法确定但会影响后续实现的内容，使用 `[NEEDS CLARIFICATION]` 标记。
5. 避免把实现方案、技术选型、代码结构提前写进 spec。

## 执行流程

### 1. 生成 feature 名称

根据需求生成简短英文 `kebab-case` 名称，作为工作区目录名。

规则：

- 2 到 6 个词
- 小写
- 连字符分隔
- 优先使用 action-noun 风格
- 保留技术术语

示例：

- `add-user-auth`
- `fix-payment-timeout`
- `order-export`

### 2. 创建工作区

如果 `specs/<feature>/` 已存在，先确认是否复用或重新命名。

若不存在，则创建：

- `specs/<feature>/`

并把该 feature 写入：

- `specs/.active`

### 3. 做范围级只读探索

在开始写 spec 前，可以做少量只读探索，回答这些问题：

- 系统现在已经具备哪些相邻能力
- 这次是新增、修改还是移除
- 涉及哪些主要模块或业务概念
- 有没有明显的范围边界或历史约束

这里的探索只为帮助界定范围，不深入实现细节。

### 4. 生成 spec.md

读取 `spec-template.md`，创建：

- `specs/<feature>/spec.md`

重点写清：

- 用户场景
- 功能需求
- 非功能要求
- 范围外内容

对未确定但重要的内容，用 `[NEEDS CLARIFICATION]` 标记。

### 5. 质量检查

在结束前至少自查：

- 范围是否清楚
- 用户场景是否足够支撑后续设计
- spec 中是否混入实现方案
- 是否存在必须尽快澄清的阻塞点

### 6. 输出结果

输出时应明确：

1. feature 名称
2. 生成的 spec 路径
3. 当前是否存在 `[NEEDS CLARIFICATION]`
4. 下一步建议：
   - 有关键歧义 → `clarify`
   - 无关键歧义 → `plan`

## 注意事项

- 不要把 spec 写成技术设计文档
- 不要为了“完整”而凭空补实现细节
- 不要一开始就陷入代码调用链分析；那是 `plan` 阶段的职责
