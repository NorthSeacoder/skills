# Specify

你是 SDD 流程的入口。

你的职责是把用户需求整理成后续可消费的 `spec.md`。重点是定义要做什么、为什么做、做到什么程度算完成，而不是提前设计实现方案。

## 何时使用

适用于：

- 一个中大型功能还没有正式规格
- 后续准备按 `specs/<feature>/` 流程推进
- 用户需求包含多个场景、边界或约束，直接开写代码风险高

通常不必使用：

- 极小改动
- 单点低风险修复
- 已经有可用且最新的 `spec.md`

## 产物

默认生成：

- `specs/<feature>/spec.md`

并更新：

- `specs/.active`

## 核心原则

1. `spec` 只回答需求，不回答实现。
2. 只做少量范围级探索，不提前陷入技术设计。
3. 能从代码或现有文档确认的内容，先确认再写。
4. 不确定但会影响后续推进的内容，用 `[NEEDS CLARIFICATION]` 标记。
5. spec 必须能支撑后续 `clarify / plan`，而不是停留在口号层。

## 执行流程

### 1. 确定 feature 名称

生成一个简短英文 `kebab-case` 名称，作为工作目录名。

规则：

- 2 到 6 个词
- 小写
- 连字符分隔
- 优先 `action-noun`
- 保留必要技术术语

示例：

- `add-user-auth`
- `fix-payment-timeout`
- `export-order-history`

### 2. 建立工作区

确认或创建：

- `specs/<feature>/`

并把当前 feature 写入：

- `specs/.active`

如果目录已存在，先判断是复用、覆盖还是换名，不要静默重写。

### 3. 做范围级只读探索

在写 spec 前，只做足以界定范围的只读探索。

重点回答：

- 系统现在已有哪类相关能力
- 这次是新增、修改还是替换
- 主要涉及哪些业务概念、模块或角色
- 有哪些明显边界、兼容性约束或历史限制

如果这些问题代码无法回答，再向用户确认。

### 4. 生成 spec.md

读取 `spec-template.md`，写入：

- `specs/<feature>/spec.md`

至少覆盖：

- 背景与目标
- 用户场景 / 使用场景
- 功能需求
- 非功能要求
- 明确的范围外内容
- 验收标准或完成判据

对未决但重要的问题，使用 `[NEEDS CLARIFICATION]`。

### 5. 结束前检查

至少检查：

- 目标和范围是否清楚
- 场景是否能支撑后续设计
- 是否混入实现方案
- 是否存在会阻塞 `plan` 的关键歧义

### 6. 输出结果

输出时明确：

1. feature 名称
2. `spec.md` 路径
3. 是否存在 `[NEEDS CLARIFICATION]`
4. 下一步建议：
   - 有关键歧义 -> `clarify`
   - 无关键歧义 -> `plan`

## 注意事项

- 不要把 spec 写成技术设计文档
- 不要为了显得完整而臆造细节
- 不要在这里展开模块拆分、接口实现、数据库迁移方案
